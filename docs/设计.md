# Mx Dev Toolkit 设计

最后更新: 2026-04-16

## 1. 项目定位

`Mx Dev Toolkit` 是一个模块化的 VS Code 开发工具箱扩展。

当前稳定能力包括:

1. `keil`
2. `selection`
3. `http_client`
4. `toast`

项目已经不再是“只有 Keil 辅助命令”的单能力插件。当前仓库同时包含扩展宿主代码、HTTP Client 的 React Webview 前端、统一 Toast 基础能力以及对应测试。

## 2. 总体架构

### 2.1 装配入口

扩展入口是 `src/extension.ts`。

当前职责非常明确:

- 创建输出通道 `mx-dev-toolkit`
- 创建统一 `ToastService`
- 装配 `registerHttpClient(context, channel, toastService)`
- 装配 `registerKeil(context, channel, toastService)`
- 装配 `registerSelection(context, toastService)`

也就是说, `extension.ts` 只负责装配与生命周期接线, 不承载具体业务细节。

### 2.2 分层关系

```text
VS Code Extension Host
├─ src/extension.ts
├─ src/keil/*
├─ src/selection/*
├─ src/http_client/*
└─ src/toast/*

HTTP Client Webview Frontend
├─ webviews/http_client/workbench/*
├─ webviews/http_client/sidebar/*
├─ webviews/http_client/shared/*
├─ webviews/http_client/styles/*
└─ webviews/http_client/tests/*

Generated Assets
├─ out/extension.js
└─ media/http_client/*
```

### 2.3 运行边界

- 扩展宿主负责命令注册、持久化、请求执行、日志和 Webview 消息桥。
- React Webview 负责工作台与侧边栏 UI 渲染、局部交互、会话态维护和宿主消息消费。
- `http_client` 当前已采用 `Webview local-first + Host service-backend` 运行边界。
- 统一提示通过 `ToastService` 调度, Webview 可见时优先走自定义 Toast, 否则回退到 VS Code 原生消息。

## 4. 模块说明

### 4.1 keil

`keil` 模块负责围绕 Keil 工程提供宿主侧能力。

关键职责:

- 解析 Keil 工程与配置
- 调用 `UV4.exe` 执行 build / rebuild / clean
- 在工作区中生成 `mx_dev.json`
- 通过输出通道输出执行日志

典型文件职责:

- `keil.ts`: 编排层, 管状态与流程组织
- `parser.ts`: 工程解析层
- `runner.ts`: 运行执行层
- `register.ts`: 命令与事件注册

### 4.2 selection

`selection` 模块是轻量编辑器增强能力。

关键职责:

- 状态栏显示当前选区行数
- 复制相对路径范围
- 复制绝对路径范围
- 统一路径格式为正斜杠, 并在 Windows 绝对路径下固定输出大写盘符

典型文件职责:

- `line_count.ts`: 选区行数状态栏
- `copy_path_range.ts`: 路径范围复制
- `register.ts`: 命令与监听装配

### 4.3 toast

`toast` 模块是跨模块共享基础设施。

关键职责:

- 暴露统一 `notify()` 入口
- 规范化 Toast 数据
- 选择最合适的可见宿主
- 在无宿主时回退到 VS Code 原生消息

关键实现:

- `service.ts`: 宿主调度、默认时长、优先级与原生回退
- `webview.ts`: 注入式 Webview Toast host 标记、样式和脚本
- `tests/service.test.ts`: 宿主优先级与回退测试

### 4.4 http_client 宿主层

`http_client` 是当前最复杂的业务模块, 由宿主层与 React Webview 前端共同组成。

宿主层职责:

- 注册命令与 HTTP Client 状态栏入口
- 维护工作台与侧边栏控制器
- 管理请求集合、环境、历史、收藏和草稿持久化
- 执行普通请求与小规模压测
- 处理 cURL 导入
- 作为工作台/侧边栏与前端之间的 service-backend 与消息桥
- 接入统一 ToastService

关键文件:

- `register.ts`: 注册命令与侧边栏 provider
- `panel.ts`: 工作台控制器, 负责工作台生命周期、状态快照、宿主命令和消息桥
- `sidebar_view.ts`: 侧边栏 provider, 负责侧边栏 Webview 生命周期与消息桥
- `store.ts`: 存储层
- `resolver.ts`: 环境变量解析
- `runner.ts`: 普通请求执行
- `load_runner.ts`: 小规模压测执行
- `curl_import.ts`: cURL 导入
- `types.ts`: 协议、视图状态与共享类型

### 4.5 http_client Webview 装载层

宿主侧仍然保留最小 Webview 装载层。

当前主路径:

- `src/http_client/webview/index.ts` 直接返回 `getReactWorkbenchHtml(...)`
- `src/http_client/sidebar_view.ts` 中的 `getSidebarHtml(...)` 直接返回 `getReactSidebarHtml(...)`
- `src/http_client/webview/react_html.ts` 负责:
  - 解析 `media/http_client/*.js` / `*.css`
  - 注入 `window.__MX_HTTP_CLIENT_BOOTSTRAP__`
  - 注入统一 Toast host 标记、样式和脚本
  - 生成 CSP 与 nonce

这意味着:

- 当前运行时主 UI 已经切换到 React
- 宿主层仍然负责 HTML 外壳、初始状态注入和资源引用
- React 前端不是独立站点, 而是被 VS Code Webview 宿主装载的双入口界面

### 4.6 http_client React 前端

当前 React Webview 前端位于 `webviews/http_client/`。

真实实现边界:

- `workbench/main.tsx`: 工作台入口
- `sidebar/main.tsx`: 侧边栏入口
- `shared/bootstrap.ts`: 读取宿主注入 bootstrap
- `shared/vscode.ts`: Webview 侧 VS Code API 适配
- `shared/workbench_model.ts` 与 `shared/sidebar_model.ts`: 共享纯逻辑模型
- `useWorkbenchController.ts` 与 `useSidebarController.ts`: 宿主消息与前端状态控制桥

当前 `workbench` 的关键事实是:

- 首屏直接消费宿主注入的 `initialState` snapshot
- 高频交互优先在同一个 Webview 内本地完成
- Host 只在结构性刷新、执行结果回推和异常恢复时发送整包 `httpClient/state`

Vite 配置事实:

- `root = webviews/http_client`
- `outDir = media/http_client`
- `appType = "custom"`
- `plugins = [react(), tailwindcss()]`
- 双入口:
  - `workbench/main.tsx`
  - `sidebar/main.tsx`

### 4.7 关于旧 `src/http_client/webview/ui/*`

这些文件当前仍存在于仓库中, 但不应继续被文档当成主运行路径。

更准确的理解是:

- 它们属于旧宿主内联 UI 路径遗留/兼容层的一部分
- 现在主运行表面已经切换为 React Webview
- 如果文档需要说明目录现状, 必须明确“存在”与“当前主运行路径”是两回事

## 5. HTTP Client 通信与运行模型

### 5.1 工作台

- 宿主控制器: `HttpClientPanelController`
- Webview HTML: `getHttpClientHtml(...) -> getReactWorkbenchHtml(...)`
- 初始化消息: `httpClient/init`
- 宿主命令转发: `httpClient/hostCommand`
- 统一 Toast 消息: `mxToast/show`

### 5.2 侧边栏

- 宿主 provider: `HttpClientSidebarProvider`
- Webview HTML: `getSidebarHtml(...) -> getReactSidebarHtml(...)`
- 初始化消息: `httpClientSidebar/init`
- 状态推送: `httpClientSidebar/state`
- 统一 Toast 消息: `mxToast/show`

### 5.3 Toast 优先级

当前宿主优先级:

1. `httpClient.panel` = `100`
2. `httpClient.sidebar` = `50`
3. VS Code 原生消息回退

## 6. 构建链

当前构建链已经分为两部分:

### 6.1 Extension host

- 入口: `src/extension.ts`
- 打包: `esbuild`
- 产物: `out/extension.js`

### 6.2 Webview frontend

- 入口: `webviews/http_client/workbench/main.tsx` 与 `webviews/http_client/sidebar/main.tsx`
- 打包: `Vite`
- 产物: `media/http_client/`

对应脚本:

- `pnpm compile:webview`
- `pnpm compile:extension`
- `pnpm compile`
- `pnpm watch:webview`
- `pnpm watch:extension`
- `pnpm watch`

当前仓库已经存在模块测试, 不应再描述为“尚未配置测试用例”。

- `pnpm test`

## 8. 开发约定

- `extension.ts` 只做装配, 不堆业务逻辑。
- 模块优先通过 `register.ts` 暴露装配入口。
- 宿主与 Webview 的边界要清晰: 宿主保业务事实源, Webview 保渲染与局部交互。
- 设计文档描述当前主路径时, 必须优先以真实入口文件为准, 不能只按历史目录推断。
- 不要手改 `out/`、`media/http_client/` 和 `eh_keil_tool/target/`。
